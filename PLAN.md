**Watch item:** Flash-tier models trade reasoning depth for speed/cost versus Pro-tier. The PRD's confidence-scoring rules (§11.7) and speculative-language detection (holdouts A4, A6 in EVALS.md) lean on exactly the kind of nuanced judgment Flash may handle less reliably than Pro. If Phase 4's `npm run eval` pass rate on those specific fixtures is weak, escalate the extraction call (only) to Gemini 2.5 Pro while keeping Flash for lighter tasks (PII tagging, drafting) — don't silently lower the confidence thresholds to compensate.

## Test infrastructure (set up in Phase 1, grows every phase)

Three layers, one command each:

1. **`npm test` — unit (Vitest).** Pure functions, no network: validators, unit-sanity check (GR-1), dedupe (GR-2), date/period logic, drafter input filter, citation parser. SQLite via better-sqlite3 `:memory:`.
2. **`npm run test:integration` — pipeline (Vitest).** Drives the real modules end-to-end against fixture data and a temp SQLite file: card state transitions, idempotency, conflict flow, PII state machine, drafter refusals. Slack/Google clients mocked at the API boundary; Bolt handlers kept thin so logic is testable without a socket.
3. **`npm run eval` — LLM fixtures.** JSON goldens in `evals/fixtures/` run through the *real* extraction and PII prompts (temperature 0), asserted structurally (item counts, flags, quote-verbatim, confidence bands) — never on exact prose. This is EVALS.md groups A and C1a/C3. Track pass rate in `evals/RESULTS.md`; a regression here blocks the phase gate.

Architecture rule that makes this possible: `src/core/` is pure logic (extraction contract, validators, delta/gap/draft engines, state machines) — fully testable offline; `src/slack/` and `src/google/` are thin adapters. Manual sandbox smoke tests only prove the adapters.

**Build notes (Jul 6 2026):** Discovered mid-phase that Sheet numbers had never flowed through the `evidence` table — Phase 1's Ledger Summary displayed Sheet values directly, bypassing the evidence/confirm pipeline entirely. Both §9.4 ("run this check before proposing") and §9.5 ("compare Slack-sourced values against Sheet values") require Sheet numbers to exist as real evidence rows, so `src/core/sheetEvidenceMapper.ts` + a new `runSheetEvidenceAndChecks` pipeline step now propose Sheet evidence (workshops completed, per-session attendance, students served) the same way Slack evidence is proposed, with GR-2 dedupe making repeat scans safe. A genuine bug was caught by testing the numeric-fallback parser against the real demo string ("Workshop 8 was completed with an attendance of 54 students.") — a naive "first number in the string" heuristic grabbed the session index (8) instead of the count (54); fixed by stripping the "Workshop N"/"Session N" reference before parsing. Also found and fixed a live regression in `src/google/sheets.ts` (missing cell-reference fields, causing a build failure) and two pre-existing test-authoring bugs in `src/core/piiDetection.ts`'s test file, both introduced by out-of-scope work from an earlier workflow run not requested as part of this phase — see session record for detail. PII masking uses a dedicated Gemini prompt (`src/llm/piiMasking.ts`) verified against the existing regex backstop before anything renders; a masking result that still trips the regex backstop is treated as unsafe and no card is posted (fail closed).

**Live-test findings (Jul 6 2026):** B1, B2, and C2 all verified end-to-end live in the sandbox, plus three more bugs caught only by live testing (all fixed): (1) masking was gated behind "was this row inserted just now," so a PII row that already existed from an earlier scan (GR-2 dedupe) could never get masked — fixed by decoupling masking into a pass over all `pii_state='detected'` rows regardless of insertion recency; (2) the same GR-2-vs-freshness bug existed for the attendance conflict check — fixed the same way; (3) the regex backstop, when verifying the LLM's *masked output* (not raw input), flagged the bare word "daughter" as unsafe even with no name/location attached — but PRD §13.6's own example output keeps that word; added `detectResidualPiiInMaskedText` (checks only literal centre names and age/minor patterns, not family-relation words) so correctly-masked text isn't rejected; (4) the "Reveal original" ephemeral message posted outside the assistant thread's visible pane (missing `thread_ts`) — fixed by threading it via `body.message.thread_ts`.

**C3/C4 closed out (Jul 6 2026):** Testing both surfaced one real architectural gap, now fixed. C3 (`evals/fixtures/c3-quasi-identifier.json`, run against the live Gemini extraction call) showed the LLM's own `pii_detected` flag came back `false` for "the tallest girl at the Kolathur centre, her father drives an auto" — a quasi-identifier with no name — but the independent regex backstop caught it (`family relation term: "father"`, `location: "Kolathur"`). That backstop wasn't actually independent at the time: `runEvidencePipeline` only ever consulted `detectPiiRegex` *after* an item was already LLM-flagged, so a case where the LLM alone said "false" would have slipped through with no PII protection at all. Fixed in `src/slack/assistant.ts` by OR-ing `item.pii_detected` with a fresh `detectPiiRegex` call on every extracted item before routing, closing the gap C3 exists to catch. C4 (roster names must never be quoted) was verified two ways: architecturally, `readAttendanceTrackerSnapshot` (`src/google/sheets.ts`) reads the Roster tab's raw rows only to search for the one row matching "unique"/"enrolled" — individual name rows are held in a local array and never returned, logged, or exposed; `sheetEvidenceMapper.ts` only ever maps aggregate counts (workshop count, per-session counts, students-served count) into evidence candidates, never row-level roster data. Empirically, querying the live `grantproof.db` for all Sheet-sourced evidence rows (`source_ref LIKE '%!%'`) showed every `claim_text`/`quote_text` is a bare number or a templated count sentence — no name or roster row ever reached the evidence table. Locked in with a new regression test in `test/unit/sheets-client.test.ts` that plants two fake student names in mock roster rows and asserts neither name appears anywhere in the parsed snapshot's serialized output. `npx tsc --noEmit` clean, 108/108 tests passing.

Fixed `verifySessionPhotos` to parse the date from the filename first (e.g. "July 18.jpg" → 2026-07-18, using the reporting period's year), falling back to `createdTime` only when the filename has no parseable date. ## Agents

**In the product: exactly one.** GrantProof is a single Slack agent (the assistant-pane app). The PRD explicitly rejects an agent framework — retriever, extractor, validators, gap detector, and drafter are modules in one pipeline, not agents. Inside that pipeline there are **three distinct LLM call types** (extraction §9.2, PII tag pass §10, drafter §12) — same API, three prompts, each with code-side validation. The read-only MCP ledger server (2 tools) is a post-Phase-5 stretch that would let *other* agents consume our evidence — never a dependency.

**In development (Claude Code subagents), per phase:**
- `tdd-guide` — start of every phase: write the phase's EVALS-mapped tests first
